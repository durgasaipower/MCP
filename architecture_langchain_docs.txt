SECTION 2: TOOLS - DETAILED DESCRIPTION
================================================================================

───────────────────────────────────────────────────────────────────────────────
TOOL 1: search_docs_by_lang_chain
───────────────────────────────────────────────────────────────────────────────

PURPOSE:
    Semantic and keyword search across the entire LangChain documentation
    knowledge base (1,377 pages). This is the RAG retrieval component.

HOW IT WORKS (Internal - RAG Pipeline):
    1. User query comes in: "how to create an agent"
    2. Query is embedded/processed for matching
    3. Searched against pre-indexed documentation chunks
    4. Top-K relevant chunks are retrieved and ranked
    5. Results formatted with titles, snippets, and page links
    6. Returned to the caller

INPUT:
    {
        "query": "string"  (required)
    }

OUTPUT:
    {
        "content": [
            {
                "type": "text",
                "text": "<markdown formatted search results with titles, snippets, links>"
            }
        ]
    }

WHEN TO USE:
    - Broad conceptual questions ("how do I...")
    - Finding tutorials, guides, examples
    - API reference lookups
    - Discovery (don't know the exact page path)


───────────────────────────────────────────────────────────────────────────────
TOOL 2: query_docs_filesystem_docs_by_lang_chain
───────────────────────────────────────────────────────────────────────────────

PURPOSE:
    Direct read-only access to all 1,377 documentation pages via a
    virtualized in-memory filesystem. Execute shell commands to explore,
    read, and search the docs.

INPUT:
    {
        "command": "string"  (required)
    }

    Supported commands:
        tree, ls, cat, head, tail, rg, grep, find, stat, wc,
        sort, uniq, cut, sed, awk, jq

    NOT supported: writes, network, process control

OUTPUT:
    {
        "content": [
            {
                "type": "text",
                "text": "<command stdout/stderr output as plain text>"
            }
        ]
    }

    Output truncated at 30KB per call. Each call is STATELESS.

WHEN TO USE:
    - Reading full page content by path
    - Exploring directory structure
    - Exact keyword/regex matching (rg)
    - Reading OpenAPI specs
    - After search, to get full page content


───────────────────────────────────────────────────────────────────────────────
RESOURCE: mintlify://skills/langchain
───────────────────────────────────────────────────────────────────────────────

    Markdown guidance on when to use:
    - LangChain: High-level agent abstractions
    - LangGraph: Low-level orchestration, memory, human-in-the-loop
    - Deep Agents: Batteries-included agent capabilities

SECTION 5: SEARCH vs FILESYSTEM - WHEN TO USE WHICH
================================================================================

┌─────────────────────────────────────┬────────────────────────────────────────┐
│  search_docs_by_lang_chain          │  query_docs_filesystem                 │
├─────────────────────────────────────┼────────────────────────────────────────┤
│  "How do I build a RAG pipeline?"   │  "cat /oss/python/langchain/rag.mdx"   │
│  Semantic understanding             │  Exact path/keyword lookup             │
│  Fuzzy matching                     │  Precise matching (regex)              │
│  Returns snippets + links           │  Returns raw file content              │
│  Good for discovery                 │  Good for reading known pages          │
│  Like Google Search                 │  Like a file explorer / terminal       │
│  Best first step                    │  Best second step (after search)       │
└─────────────────────────────────────┴────────────────────────────────────────┘

Typical workflow:
    1. search_docs_by_lang_chain("RAG tutorial")
       → Returns: "Found: /oss/python/langchain/rag.mdx ..."

    2. query_docs_filesystem("cat /oss/python/langchain/rag.mdx")
       → Returns: Full page content with all code examples

SECTION 8: HOW TO BUILD SOMETHING SIMILAR
================================================================================

To replicate this for your own documentation:

    1. CONTENT LAYER
       - Store docs as .mdx or .md files in a Git repo
       - Organize in logical folder hierarchy
       - Each file = one documentation page

    2. SEARCH/RAG LAYER (for search tool)
       - Chunk documents into segments (by heading, paragraph, etc.)
       - Create embeddings (OpenAI, Cohere, etc.)
       - Store in vector DB (ChromaDB, Pinecone, Weaviate, etc.)
       - Re-index on every git push / deploy
       - Implement search: query → embed → similarity search → rank → return

    3. FILESYSTEM LAYER (for filesystem tool)
       - Load all .mdx files into memory as virtual filesystem
       - Implement command parser for: tree, ls, cat, head, rg, find, etc.
       - Return stdout-like output
       - Keep it read-only and stateless

    4. MCP SERVER LAYER
       - Use MCP SDK:
           Python:     from mcp.server.fastmcp import FastMCP
           TypeScript: import { McpServer } from "@modelcontextprotocol/sdk"
       - Define 2 tools with proper schemas
       - Serve over Streamable HTTP transport

    5. DEPLOYMENT
       - Host on Vercel / Cloudflare Workers / AWS Lambda
       - Put behind CDN
       - Single /mcp endpoint
       - No auth needed for public docs

    Minimal Python example:
        from mcp.server.fastmcp import FastMCP
        mcp = FastMCP("My Docs Server")

        @mcp.tool()
        def search_docs(query: str) -> str:
            # Your RAG retrieval logic here
            results = vector_db.similarity_search(query, k=5)
            return format_results(results)

        @mcp.tool()
        def query_filesystem(command: str) -> str:
            # Your virtual filesystem logic here
            return execute_virtual_command(command)

        mcp.run(transport="streamable-http", port=8000)
